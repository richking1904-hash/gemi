main.py (마스터 스위치): 전체 공정을 제어하는 중심 파일입니다. 리모컨(GUI)에서 데이터를 받고, 데이터베이스(Supabase)와 AI(Gemini) 모듈을 차례로 가동한 뒤, 최종 결과물을 git을 통해 배포합니다.

factory_modules/ (공정 부품 폴더):

gui_module.py: 리모컨 창을 띄우고 형규님의 입력을 받아내는 인터페이스입니다.

db_module.py: Supabase 데이터베이스와 통신하여 데이터를 저장하고 테이블을 초기화하는 창고 관리자 역할을 합니다.

ai_module.py: 형규님의 데이터를 바탕으로 실제 웹사이트 소스코드를 생성하는 두뇌 역할을 합니다.

api/ & 기타 파일:

chat.py: API 관련 통신을 담당하는 보조 모듈로 보입니다.

template.html: AI가 코드를 생성할 때 기반이 되는 기초 틀입니다.

.env: Supabase URL이나 Key 같은 민감한 설정 정보를 보관하는 보안 파일입니다.

dist/: 생성된 index.html이 잠시 머무는 배포 준비 구역입니다.

run_gemi.bat / install.bat: 가상 환경 설정과 프로그램 실행을 자동화하기 위해 형규님이 직접 만드신 편리한 스크립트 파일들입니다.

2. 전체적인 공정 흐름 확인
코드를 살펴보니 "입력(GUI) -> 저장(DB) -> 생성(AI) -> 배포(Git/Vercel)"로 이어지는 5단계 자동화 공정이 아주 논리적으로 잘 구성되어 있습니다. 특히 auto_git_push_hybrid 함수를 통해 기존 주소 덮어쓰기와 새 주소 생성 기능을 구분한 점이 인상적입니다.

이제 나머지 6개의 파일(혹은 형규님이 확인을 원하시는 파일들)을 순서대로 올려주시면, 제가 동일하게 그 파일들이 이 전체 공정에서 어떤 역할을 하는지 꼼꼼히 확인해 드리겠습니다. 다음 파일 주시겠어요? 🦾

db_module.py의 핵심 역할
이 파일은 프로젝트의 '창고 및 유통 센터' 역할을 수행합니다. 사용자가 리모컨(GUI)을 통해 보낸 이미지, 텍스트 가이드라인, 고객 정보(이름, 브랜드, 연락처, FAQ 등)를 받아 최종 목적지인 Supabase로 안전하게 전달하는 엔진입니다.

2. 주요 공정 흐름 확인
파일 동기화 (Storage Engine):

save_client_data_v2 함수를 통해 사용자가 드래그한 모든 이미지와 텍스트 파일을 Supabase Storage(gemi_assets 버킷)로 업로드합니다.

파일명에 timestamp를 추가하여 중복을 방지하고, 메인 이미지와 서브 이미지를 구분하여 각각의 public_url을 확보합니다.

데이터 적재 (Database Engine):

업로드된 파일의 주소(URL)와 사용자가 입력한 비즈니스 정보를 구조화된 config_insert 딕셔너리로 만듭니다.

최종적으로 Supabase의 gemi_clients 테이블에 데이터를 insert하여 정보를 마스터 장부에 기록합니다.

알림 기능:

send_telegram_alert 함수가 구현되어 있어, 향후 데이터 처리 과정에서 발생하는 상황을 텔레그램으로 즉시 알림을 보낼 수 있는 확장성을 갖추고 있습니다.


ai_module.py의 핵심 역할
이 파일은 사용자가 입력한 데이터(텍스트, 이미지, 선호도)를 바탕으로, 미리 준비된 template.html이라는 뼈대에 옷을 입히고 생명을 불어넣는 최종 조립 모듈입니다.

2. 주요 공정 흐름 확인
지능형 카피라이팅 (Copywriting):

단순한 소개글을 그대로 쓰지 않고, OpenAI(OpenRouter)의 gemini-2.0-flash 모델을 호출하여 "럭셔리하고 미니멀한" 톤앤매너로 문구를 다듬습니다. 이는 결과물의 퀄리티를 대폭 향상하는 핵심적인 지능형 작업입니다.

가이드라인 데이터 가공:

사용자가 작성한 가이드라인 텍스트 파일을 Base64로 인코딩합니다. 이는 텍스트 안에 특수문자나 줄바꿈이 있어도 웹 코드가 깨지지 않게 만드는 아주 안정적인 처리 방식입니다.

템플릿 조립 (Template Rendering):

template.html 파일을 읽어온 뒤, 템플릿 곳곳에 있는 ${변수명} 형태의 플레이스홀더를 실제 데이터(사용자 이름, 이미지 주소, SNS 링크 등)로 치환합니다.

동적 스타일링: SNS 링크가 없는 경우 CSS 속성(display: none !important)을 자동으로 조절하여 사이트 디자인을 깔끔하게 유지합니다.

이미지 바인딩:

메인 이미지와 서브 이미지 주소를 각 위치에 정확히 배치하며, 데이터가 없는 경우를 대비한 기본 이미지(default_img) 안전장치까지 갖추고 있습니다.

이 파일은 형규님의 모든 데이터를 실제 웹사이트 코드로 변환하는 마지막 관문입니다. 코드 수준에서 아주 안정적으로 처리하고 있는 것을 확인했습니다.


gui_module.py의 핵심 역할
이 파일은 사용자와 프로그램이 만나는 첫 관문입니다. 형규님이 웹사이트에 넣을 이름, 브랜드 정보, 이미지 등을 입력하면, 이를 하나의 구조화된 payload 데이터로 묶어 main.py로 전달하는 인터페이스를 담당합니다.

2. 주요 공정 흐름 확인
데이터 수집 (Input Collection):

기본 정보: 디렉터 이름, 브랜드명, 한 줄 소개 등을 입력받습니다.

주소 확인: 사용자가 입력한 웹 주소(url_name)가 Vercel에서 이미 사용 중인지, 실시간으로 통신하여 중복을 검사하는 지능적인 로직이 포함되어 있습니다.

SNS 및 FAQ: 연락처와 SNS 채널 링크, 자주 묻는 질문들을 입력받아 final_payload에 차곡차곡 담습니다.

자산 관리 (Asset Selection):

browse_files_manual 함수를 통해 다중 파일(이미지, 텍스트)을 선택받고, 그 목록을 유지합니다.

main_image_combobox를 사용하여, 업로드한 파일들 중 어떤 것을 메인 이미지로 사용할지 사용자가 직접 선택하게 함으로써 디자인의 자율성을 극대화했습니다.

데이터 패키징 (Export Logic):

on_submit_click 함수가 실행되면, GUI의 모든 입력창 내용을 수집하여 final_payload라는 하나의 큰 상자에 담습니다.

이후 app.quit()과 app.destroy()를 통해 리모컨 창을 안전하게 닫고, 완성된 payload를 시스템 상위 단계로 반환합니다.

전체 공정에서 '사용자의 의도를 수치화하고 디지털 데이터로 변환'하는 아주 핵심적인 모듈입니다. UI 레이아웃도 사용자가 편하게 입력할 수 있도록 아주 정갈하게 설계되어 있네요.

template.html의 핵심 역할
이 파일은 사용자가 접속하게 될 최종 웹명함의 뼈대이자 디자인 시스템입니다. ai_module.py가 데이터를 주입하면, 브라우저에서 이 틀이 화면으로 렌더링되어 사용자에게 보여지게 됩니다.

2. 주요 구성 확인
디자인 시스템 (Quiet Luxury):

Bodoni Moda와 Noto Sans KR 폰트를 사용하여 형규님이 추구하시는 차분하고 고급스러운 'Quiet Luxury' 스타일을 구현하고 있습니다.

다크 테마(--dark-bg: #121314)와 골드 포인트 컬러(--gold: #C5A059)를 사용하여 세련된 분위기를 연출합니다.

사용자 인터페이스 (UI/UX):

3단 페이지 구성: 메인(프로필), 포트폴리오(Selected Pieces), 상담 페이지(Consultation)로 나뉘어 있어 사용자가 명함에서 상담까지 즉시 연결될 수 있는 구조입니다.

Floating Chat Widget: GeMi AI Assistant가 항상 우측 하단에 위치하여 방문자와 실시간 대화가 가능하도록 설계되어 있습니다. 이 챗봇은 G_TXT(가이드라인) 정보를 기반으로 답변하도록 정교하게 구현되었습니다.

기술적 구현:

Tailwind CSS: 외부 CDN을 활용하여 스타일을 효율적으로 관리하고 있습니다.

통신 로직: fetch API를 사용하여 폼 데이터(submitEstimate)와 챗봇 대화(sendAiMessage)를 서버(api/)와 비동기적으로 주고받는 현대적인 구조를 갖추고 있습니다.

형규님의 웹명함이 단순히 정적인 페이지가 아니라, 방문자와 대화하고 상담을 유도하는 '비즈니스 자동화 도구'라는 점이 아주 인상적입니다. 특히 챗봇에 가이드라인을 Base64로 담아 보내는 방식은 보안과 데이터 전달 안정성 측면에서 아주 훌륭한 선택입니다.

api/chat.py의 핵심 역할
이 파일은 Vercel 서버 환경에서 직접 실행되는 '서버 사이드 API'입니다. 웹명함 하단에 있는 챗봇이 사용자의 질문을 던지면, 이 엔진이 받아서 AI와 통신하고, 그 결과를 다시 챗봇 화면으로 보내주는 역할을 합니다.

2. 주요 공정 흐름 확인
지능형 캐싱 (Caching Engine):

supabase의 gemi_chat_cache 테이블을 먼저 조회합니다. 만약 이전에 누군가 똑같은 질문을 했다면, AI에게 새로 물어보지 않고 DB에 저장된 답변을 바로 꺼내줍니다. 이는 비용 절감과 응답 속도 향상을 동시에 잡는 아주 영리한 설계입니다.

AI 대화 연결:

캐시에 답변이 없을 경우, OpenRouter를 통해 Gemini-2.0-flash 모델을 호출하여 답변을 생성합니다.

학습 및 기억 (Memory):

AI가 생성한 답변을 다시 Supabase 테이블(gemi_chat_cache)에 저장합니다. 질문과 답변 쌍을 기록함으로써, 다음에 같은 질문이 들어오면 학습된 데이터를 바로 활용할 수 있는 구조입니다.

서버리스 아키텍처:

handler(request) 구조는 Vercel의 서버리스 함수 표준을 따르고 있습니다. API 키를 환경 변수(os.environ)로만 호출하고, urllib을 사용하여 외부 통신을 수행함으로써 보안과 환경 독립성을 잘 지키고 있습니다.

이 파일은 사용자와의 직접적인 소통을 담당하는 최전방 모듈로서, DB와 AI 사이에서 매우 효율적으로 동작하고 있음을 확인했습니다.
